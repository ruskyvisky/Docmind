# рҹ§  DocMind

**DocMind** is a fully local, privacy-first **Retrieval-Augmented Generation (RAG)** system that lets you upload PDF documents and ask questions about them in natural language. All processing вҖ” including embedding generation and LLM inference вҖ” happens on your own machine using [Ollama](https://ollama.com/). No data ever leaves your environment.

## вҡҷпёҸ How It Works

DocMind implements a classic RAG pipeline with a **Hybrid Search** layer on top:

```
PDF Upload
    в”Ӯ
    в–ј
[BullMQ Queue]          вҶҗ Async job queuing, non-blocking upload
    в”Ӯ
    в–ј
[Worker: PDF Parsing]   вҶҗ pdf-parse extracts raw text
    в”Ӯ
    в–ј
[Text Chunking]         вҶҗ Sliding window (500 chars, 50 overlap)
    в”Ӯ
    в–ј
[Embedding Generation]  вҶҗ nomic-embed-text via Ollama (768-dim vectors)
    в”Ӯ      вҶ‘
    в”Ӯ   [Redis Cache]   вҶҗ SHA-256 keyed cache вҶ’ avoids re-embedding
    в”Ӯ
    в–ј
[PostgreSQL]            вҶҗ Stores chunks + pgvector embeddings + tsvector FTS
    в”Ӯ
    в–ј
[/ask endpoint]
    в”Ӯ
    в”ңв”Җ Vector Search  (pgvector cosine similarity, HNSW index)
    в”ңв”Җ Full-Text Search (PostgreSQL tsvector + GIN index)
    в””в”Җ Hybrid Fusion  (min-max normalized, weighted: 70% vector + 30% FTS)
         в”Ӯ
         в–ј
    [LLM: llama3.1:8b] вҶҗ Context-injected prompt, streamed response via SSE
```

## вңЁ Features

| Feature | Details |
|---|---|
| **PDF Ingestion** | Upload PDFs via REST API; processed asynchronously in the background |
| **Async Queue** | BullMQ-powered job queue with exponential backoff retry (up to 3 attempts) |
| **Smart Chunking** | Sliding window chunking (500 chars, 50-char overlap) with section-aware splitting |
| **Vector Embeddings** | 768-dimensional vectors generated by `nomic-embed-text` via Ollama |
| **Hybrid Search** | Combines semantic (pgvector cosine) and keyword (PostgreSQL FTS) results |
| **Score Normalization** | Min-max normalization ensures fair comparison between vector and text scores |
| **Relevance Threshold** | Responses are blocked when `hybrid_score < 0.30` to prevent hallucination |
| **Embedding Cache** | Redis caches embeddings by SHA-256 hash вҖ” both single and batch (`MGET`/`MSET`) |
| **Streaming Responses** | LLM answers stream token-by-token via Server-Sent Events (SSE) |
| **Fully Local** | Zero cloud dependencies; all AI runs via local Ollama instance |
| **Dockerized** | One-command deployment with `docker compose up` |
| **Concurrency Control** | Worker processes 2 PDFs simultaneously; others wait in queue |

| Service | Image | Role |

| **Redis** | `redis:alpine` | Job queue broker + embedding cache |

### Database Extensions & Indexes
- **pgvector** вҖ” stores and queries 768-dim float vectors
- **HNSW index** вҖ” approximate nearest neighbor search for fast cosine similarity
- **GIN index** вҖ” inverted index for PostgreSQL full-text search on `tsvector` column
- **Trigger** вҖ” auto-populates `search_vector` column on every `INSERT`/`UPDATE`

## рҹҡҖ Installation & Setup

### 1. Clone the repository

```bash
git clone https://github.com/your-username/docmind.git
cd docmind
```

### 2. Start all services

```bash
docker compose up --build
```

This will:
- Start PostgreSQL with `pgvector` and auto-run `init.sql` (creates tables, indexes, and the trigger)
- Start Redis
- Build and start the Node.js application

The app will be available at **http://localhost:3000**

### 3. Verify Ollama is reachable

The app connects to Ollama via `http://host.docker.internal:11434`. Make sure Ollama is running on your machine:

```bash
ollama serve   # if not already running
```

**Response:**

### Ask a question (Server-Sent Events)

```bash
curl -X POST http://localhost:3000/ask \
  -H "Content-Type: application/json" \
  -d '{"question": "What are the main findings of the report?"}'
```

The response streams as SSE events:

```
event: metadata
data: {"question":"...","searchMode":"hybrid","bestScore":0.8543,"retrievedChunks":[...]}

event: token
data: {"token":"The"}

event: token
data: {"token":" main"}

event: token
data: {"token":" findings..."}

event: done
data: {"fullAnswer":"The main findings...", "done": true}
```

### Using the Web UI

Open `index.html` directly in your browser (or serve it statically) and use the form to upload documents and ask questions with a real-time streaming interface.

## рҹ“Ў API Reference

| Method | Endpoint | Description |
|---|---|---|
| `POST` | `/upload` | Upload a PDF file (`multipart/form-data`, field: `file`) |
| `GET` | `/documents/:id/status` | Poll processing status for a document |
| `POST` | `/ask` | Ask a question; streams SSE response |

### SSE Event Types (`/ask`)

| Event | Payload | Description |
|---|---|---|
| `metadata` | `{ question, searchMode, bestScore, retrievedChunks, questionCacheHit }` | Search metadata before answer begins |
| `token` | `{ token }` | A single streamed text token from the LLM |
| `done` | `{ fullAnswer, done: true }` | Full answer after stream completes |
| `error` | `{ error, details }` | Error during streaming |

## рҹ§© Design Decisions

### Why Hybrid Search?
Pure vector search excels at semantic similarity but can miss exact keyword matches. Full-text search catches keywords but doesn't understand meaning. Combining both вҖ” with **min-max normalization** and a **70/30 weighted score** вҖ” gives more robust retrieval across diverse query types.

### Why BullMQ + Redis?
PDF processing (parsing вҶ’ chunking вҶ’ batch embedding via Ollama) is CPU and I/O intensive. Doing it synchronously in the request handler would cause timeouts on large PDFs. BullMQ decouples ingestion from the HTTP layer, provides retry logic with exponential backoff, and allows controlling concurrency (2 simultaneous jobs).

### Why Redis for Embedding Cache?
Generating an embedding via Ollama takes ~50вҖ“200ms per chunk. Re-processing the same content (e.g., uploading a document twice, or re-indexing) would be wasteful. SHA-256 hashing each chunk's text as a cache key means identical content is embedded only once. Batch `MGET`/`MSET` operations minimize round-trips.

### Why HNSW over IVFFlat?
HNSW (Hierarchical Navigable Small World) provides better query-time performance at the cost of slightly higher build time and memory. For a document Q&A use case where search latency matters more than index build speed, HNSW is the right trade-off.

### Why host-native Ollama instead of Dockerized?
Running Ollama in a Docker container would require re-downloading large model files. By connecting to the host machine's Ollama instance via `host.docker.internal`, models are shared between development and the containerized app вҖ” no duplication, no extra setup.
